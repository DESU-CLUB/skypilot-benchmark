### Root cause (`fix_explanation.txt`)
SkyPilot materialises `file_mounts` **before** running `setup`. The broken config mounted `s3://ml-configs/job-config` directly onto `/home/user/training-code`, pre-populating that directory. When `setup` later called `git clone … /home/user/training-code` it found a non-empty directory and aborted with "directory not empty".

---

### What was broken (`broken_job.yaml`)
```yaml
file_mounts:
  /home/user/training-code: s3://ml-configs/job-config   # ← collision

setup: |
  git clone https://github.com/myorg/training-code /home/user/training-code  # ← fails
```
The mount target and the clone target are the **same path**.

---

### How it was fixed (`fixed_job.yaml`)
Two changes:

| Concern | Change |
|---|---|
| **Mount path** | S3 bucket remounted to `/home/user/job-config` (a fresh, dedicated directory) |
| **Symlink** | `setup` adds `ln -sf /home/user/job-config /home/user/training-code/config` so code inside the repo can still reach config files at a stable relative path |

```yaml
file_mounts:
  /home/user/job-config: s3://ml-configs/job-config   # ← no collision

setup: |
  git clone https://github.com/myorg/training-code /home/user/training-code  # ← succeeds
  cd /home/user/training-code && pip install -r requirements.txt
  ln -sf /home/user/job-config /home/user/training-code/config

run: |
  python /home/user/training-code/train.py              # ← unchanged
```

All original requirements are preserved: the S3 configs are still available on the instance, the repo is cloned and installed, and `train.py` is executed via the same `run` command.
