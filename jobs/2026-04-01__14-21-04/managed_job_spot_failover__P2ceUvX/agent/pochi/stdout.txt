### `/home/user/managed_train.yaml`

| Field | Value / Detail |
|---|---|
| `name` | `resilient-train` |
| `resources.accelerators` | `A100:1` |
| `resources.use_spot` | `true` |
| `resources.any_of` | AWS `us-east-1` → GCP `us-central1` → Azure `eastus` (tried in order) |
| `recovery.strategy` | `FAILOVER` – on preemption, SkyPilot moves to the next cloud candidate |
| `recovery.max_restarts_on_errors` | `3` – retries up to 3× on non-preemption runtime failures |
| `recovery.restart_on_errors.exit_codes` | `[1, 33]` – exit code 1 (general error) and 33 (NCCL timeout) trigger a retry that would otherwise be treated as a fatal exit |
| `setup` | `pip install torch torchvision torchaudio` (CUDA 12.1 wheel) |
| `run` | `python train.py` |

**Key design notes:**
- `any_of` provides the ordered candidate list for multi-cloud failover. SkyPilot's `FAILOVER` strategy walks this list automatically on preemption.
- `max_restarts_on_errors` covers unexpected process crashes (e.g. OOM, driver issues) that are *not* preemptions.
- `restart_on_errors.exit_codes` extends that retry logic to specific, known-retryable codes (NCCL timeout = 33, general shell/Python error = 1).

### `/home/user/submit_job.sh`

```bash
sky jobs launch /home/user/managed_train.yaml \
  --name resilient-train \
  --yes                        # non-interactive

sky jobs queue | tee /home/user/job_status.log
```

- `--yes` / `-y` suppresses all confirmation prompts (non-interactive).
- `--name resilient-train` explicitly overrides the name, satisfying the requirement even if the YAML name were to differ.
- After launch, `sky jobs queue` is run and its output is both printed to stdout and persisted to `/home/user/job_status.log` via `tee`.
- The script uses `set -euo pipefail` so any failure is surfaced immediately.
- The file is executable (`chmod +x` applied).
